# Memory Operations in IR

Caustic's IR uses explicit load/store instructions for all memory access. Variables are accessed by their stack frame offset (for locals) or by symbol name (for globals). Structs larger than 8 bytes are manipulated by address rather than by value.

## IR_LOAD (20) -- Load from Memory

Reads a value from memory into a virtual register. The source can be either a vreg containing an address or an immediate stack offset.

### Vreg Source (Pointer Dereference)

When `src1.op_type == OP_VREG`, the vreg holds a pointer to the data:

```
IR_LOAD dest, V(addr)
```

Codegen loads the address into r15 and reads from `[r15]` with the appropriate width:
- 8 bytes: `mov r15, QWORD PTR [r15]`
- 4 bytes: `movsxd r15, DWORD PTR [r15]` (signed) or `mov r15d, DWORD PTR [r15]` (unsigned)
- 2 bytes: `movsx r15, WORD PTR [r15]` (signed) or `movzx r15, WORD PTR [r15]` (unsigned)
- 1 byte: `movsx r15, BYTE PTR [r15]` (signed) or `movzx r15, BYTE PTR [r15]` (unsigned)

### Immediate Source (Local Variable)

When `src1.op_type == OP_IMM`, the immediate is the variable's offset from rbp:

```
IR_LOAD dest, I(offset)
```

Codegen reads directly from `[rbp - offset]` with the appropriate width.

### Type Information

The `cast_to` field carries a pointer to the value's `Type` struct, used to determine:
- **Load width**: 1, 2, 4, or 8 bytes based on `type.size`
- **Sign extension**: signed types use `movsx`/`movsxd`, unsigned types use `movzx`/zero-extending `mov`

### Large Values

For types larger than 8 bytes (structs, arrays), the IR generator does not emit an `IR_LOAD`. Instead, `gen_expr` for identifiers, members, and indices returns the address vreg directly when `ty.size > 8`. The caller is responsible for using this address with `IR_COPY` or passing it to functions.

## IR_STORE (21) -- Store to Memory

Writes a value from a vreg to a memory location. Like IR_LOAD, the destination can be either a vreg (pointer) or an immediate (stack offset).

### Vreg Destination (Pointer Store)

```
IR_STORE V(addr), V(value)
```

Codegen:
- 8 bytes: `mov QWORD PTR [r15], rax`
- 4 bytes: `mov DWORD PTR [r15], eax`
- 2 bytes: `mov WORD PTR [r15], ax`
- 1 byte: `mov BYTE PTR [r15], al`

### Immediate Destination (Local Variable)

```
IR_STORE I(offset), V(value)
```

Codegen writes directly to `[rbp - offset]`.

For types larger than 8 bytes stored to a stack offset, codegen performs a multi-word copy using a loop of `mov` instructions in 8/4/2/1 byte chunks.

### Type Information

The `cast_to` field determines the store width, matching the behavior described for IR_LOAD.

## IR_COPY (35) -- Struct/Block Copy

Copies `n` bytes from a source address to a destination address. Used for struct assignment, struct parameter passing, and SRET writes.

```
IR_COPY V(dest_addr), V(src_addr), I(byte_count)
```

Codegen:
```asm
mov rdi, <dest_addr>
mov rsi, <src_addr>
mov rcx, <byte_count>
cld
rep movsb
```

This is a byte-by-byte copy, which is simple but not optimal for large structs. The `cld` instruction ensures the direction flag is clear (forward copy).

## IR_ADDR (36) -- Address of Local

Computes the address of a local variable given its stack frame offset.

```
IR_ADDR dest, I(offset)
```

Codegen:
```asm
lea r15, [rbp - offset]
```

Used when:
- Taking the address of a local variable (`&x`)
- Accessing a struct member (base address computation)
- Initializing large local variables (structs, arrays) that require address-based stores

## IR_ADDR_GLOBAL (37) -- Address of Global

Computes the address of a global variable or string constant by symbol name.

```
IR_ADDR_GLOBAL dest
  global_name = "variable_name"
  is_extern_global = 0 or 1
```

Codegen:
- Non-extern: `lea r15, [rip + variable_name]` (PC-relative addressing)
- Extern: `mov r15, QWORD PTR [rip + variable_name]` (GOT-relative load for external symbols)

Also used for string literal references, where the global name is the `.LCN` label generated by `register_string`.

## IR_GET_ALLOC_ADDR (38) -- Stack Allocation Address

Returns the address of a dynamically-sized stack allocation area. This is used for:
- SRET return buffers (temporary space to receive large struct returns)
- Enum payload construction (temporary space to build tagged union values)

```
IR_GET_ALLOC_ADDR dest, I(offset)
```

The actual stack location is computed by codegen as:
```
rbp - (arg_spill_base + stack_slots * 8 + offset)
```

Where:
- `arg_spill_base` is the size of local variables
- `stack_slots * 8` is the space used by register spill slots
- `offset` is the instruction's immediate operand

The allocation size is tracked in `IRFunction.alloc_stack_size` and accumulated during IR generation whenever SRET buffers or enum payloads are needed. Alignment to 16 bytes is applied after each allocation.

## IR_SET_CTX (39) -- Set Context Register

Prepares the r10 register before a function call or syscall, typically with the address of the destination buffer for struct returns.

```
IR_SET_CTX V(addr)     -- mov r10, addr_vreg
IR_SET_CTX I(0)        -- xor r10, r10 (clear)
```

This allows the called function to know where to write its result without an extra argument slot.

## Address Generation (gen_addr)

The `gen_addr` function in the IR generator computes memory addresses for lvalue expressions:

- **Identifiers**: Local variables use `IR_ADDR` with the var_offset. Global variables use `IR_ADDR_GLOBAL` with the symbol name.
- **Pointer dereference** (`*ptr`): The pointer expression is evaluated normally, yielding the address directly.
- **Array indexing** (`arr[i]`): For arrays, the base address is computed via `gen_addr`. For pointers, the pointer value is loaded. The index is multiplied by the element size and added to the base.
- **Struct member access** (`s.field`): The base address is computed (via gen_addr for values, gen_expr for pointers). The field offset is added to produce the member address.

This function is distinct from `gen_expr`, which loads values. An assignment like `x = 5` calls `gen_addr(x)` for the left side and `gen_expr(5)` for the right, then emits `IR_STORE`.
